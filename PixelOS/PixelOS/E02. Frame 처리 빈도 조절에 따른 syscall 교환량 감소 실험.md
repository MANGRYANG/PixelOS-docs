# 실험 보고서: [Frame 처리 빈도 조절에 따른 syscall 교환량 감소 실험]

## 실험 개요

- 실험 번호 : 02
- 관련 프로젝트 : `PixelOS`
- 실험 대상 모듈 : `app/pong.c`

***

# 서론
## 실험 목적

> 본 실험의 목적은 `PONG_FRAME_TICKS` 값을 조절하여 Pong의 프레임 처리 빈도를 낮추었을 때, OS와 게임 애플리케이션 사이에서 발생하는 syscall 교환량이 감소하는지 확인하는 것이다.

## 실험 범위

### 포함 범위

- `PONG_FRAME_TICKS` 값 변경
- 프레임 처리 빈도 변화 관찰
- `P`, `F`, `B`, `K`, `T` syscall count 비교
- 게임 플레이 관련 체감 변화 관찰

### 제외 범위

- dirty rectangle 렌더링
- 배경 layer 캐싱
- 정적 UI 캐싱
- 공, 패들, 점수 렌더링 방식 변경
- 그래픽 품질 변경
- PS/2 입력 계층 수정

***

# 본론
## 실험 가설

> `PONG_FRAME_TICKS` 값을 증가시키면 **Pong의 프레임 처리 빈도가 감소하고, 이에 따라 프레임 단위로 발생하는 OS와 게임 간의 syscall 교환량도 감소**할 것이다.
>
> 본 실험은 프레임 자체의 내부 비용을 줄이는 최적화가 아니라 **프레임의 처리 빈도를 낮추어 전체 syscall 교환량을 줄이는 최적화**이므로, 프레임에 대한 syscall의 비율인 `F/P`, `B/P`, `K/P`, `T/P`는 변화가 없을 것으로 예상된다.

| 항목    | 내용                                                            |
| ----- | ------------------------------------------------------------- |
| 독립 변수 | `PONG_FRAME_TICKS` 수치                                         |
| 종속 변수 | `P`, `F`, `B`, `K`, `T` syscall count/sec, 프레임당 syscall 호출 횟수 |
| 통제 변수 | 창 크기, 공 속도, paddle 크기, 중앙선 개수/위치/색상, 입력 조건, 측정 시간 등           |
| 예상 결과 | `PONG_FRAME_TICKS = 2u`에서 초당 syscall 호출 횟수가 절반 수준으로 감소한다.     |

***

## 기존 구현에서의 문제점  
  
> 기존 구조에서는 `PONG_FRAME_TICKS = 1u` 기준으로 매 Tick마다 프레임 처리가 수행된다.
> 이로 인해 입력 polling, game update, rendering, present가 초당 약 100회 반복된다.
  
***
  
## 적용할 최적화 기법  
  
> 적용할 최적화 기법은 **프레임 처리 빈도 조정**이다. 
> `PONG_FRAME_TICKS` 값을 증가시켜 프레임이 처리되는 주기를 늘리고, 초당 syscall 호출량의 변화를 관찰한다.

# 결론

## 실험 결론

> `PONG_FRAME_TICKS` 값을 증가시키면 Pong의 프레임 처리 빈도가 감소하고, 이에 따라 OS와 게임 간 syscall 호출량도 감소한다.

## Overview

> 이번 실험은 **Frame Budget 관점과 연결**된다.
> 
> `PONG_FRAME_TICKS = 1u` 조건에서는 PIT 100Hz 기준 약 10ms마다 프레임을 처리하므로 목표 frame budget은 약 10ms이다.
> 반면 `PONG_FRAME_TICKS = 2u` 조건에서는 약 20ms마다 프레임을 처리하므로 목표 frame budget이 더 커지고, 초당 처리해야 하는 프레임 수는 줄어든다.
> 
> 실험 결과 `P`, `F`, `B`, `K`, `T`와 같이 프레임 처리에 종속된 syscall 호출 횟수가 절반 수준으로 감소하였다. 
> 이는 **목표 Frame budget을 크게 설정하면**(더 낮은 frame 처리 빈도를 선택하면) OS와 게임 간의 초당 syscall 교환량을 줄여 **성능 부담을 낮출 수 있음**을 보여준다.
> 
> 다만 Frame budget이 증가한다는 것은 화면을 더 드물게 갱신한다는 의미이기도 하므로, **시각적 부드러움이나 입력 반응성 측면에서는 불리하게 작용할 수 있다.**
> 
> 따라서 Frame budget 증가는 무조건적인 개선 사항이 아니라, 성능 부담과 사용자 체감 품질 사이의 trade-off로 해석해야 한다.
