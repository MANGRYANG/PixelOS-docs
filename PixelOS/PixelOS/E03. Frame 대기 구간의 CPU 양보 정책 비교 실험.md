- 실험 번호 : 03
- 관련 프로젝트 : `PixelOS`
- 실험 대상 모듈 : `app/pong.c`

# 서론
## 실험 목적

> 프레임 처리 시점이 아닐 때 게임 루프가 CPU를 계속 점유하는 busy wait 방식과, Sleep을 통해 OS에 실행을 양보하는 방식 간의 효율성을 비교한다.

## 실험 범위

### 포함 범위

- `app_sleep(1)`을 사용하는 frame 대기 방식
- `app_sleep(1)`을 제거한 busy wait 방식
- `G(SYS_GET_TICKS)`와 `S(SYS_SLEEP)` 값의 변화 관측
- `F`, `B`, `K`, `T`가 frame 처리 빈도에 의해 유지되는지 확인
- 프레임 사이 대기 구간에서 OS에 실행을 양보하는지 비교

### 제외 범위

- dirty rectangle 렌더링
- 배경 layer 캐싱
- 정적 UI 캐싱
- 공, 패들, 점수 렌더링 방식 변경
- 그래픽 품질 변경
- PS/2 입력 계층 수정

# 본론
## 실험 가설

> `SYS_SLEEP`을 사용하는 경우 프레임 처리 시점이 아닐 때 Pong은 OS에 실행을 양보하므로 `S(SYS_SLEEP)` 호출이 발생하고, `G(SYS_GET_TICKS)` 호출 수는 비교적 안정적으로 유지될 것이다.
> 
> 반면 busy wait 방식에서는 프레임 처리 시점이 아닌 시점에도 게임 루프가 계속 반복되며 `SYS_GET_TICKS`를 호출하므로, `S`는 0이 되고 `G`는 크게 증가할 것이다.
> 
> 단, 실제 프레임 처리는 여전히 `elapsed_ticks < PONG_FRAME_TICKS` 조건에 의해 제한되므로 `S`, `G`를 제외한 시스템 콜 호출 횟수는 크게 변화하지 않을 것으로 예상된다.

| 항목    | 내용                                                                           |
| ----- | ---------------------------------------------------------------------------- |
| 독립 변수 | 프레임 대기 방식 (Sleep 사용 방식 / busy wait 방식)                                       |
| 종속 변수 | `G`(`SYS_GET_TICKS`), `S`(`SYS_SLEEP`), 프레임당 syscall 호출 횟수                   |
| 통제 변수 | `PONG_FRAME_TICKS`, 창 크기, 공 속도, 입력 조건, 측정 시간                                 |
| 예상 결과 | busy wait 방식에서는 `S`가 0이 되고 `G` 값이 크게 증가한다.<br>프레임당 syscall 호출 횟수에는 큰 변화가 없다. |

## 기존 구현에서의 관찰 지점
  
> 기존 tick 기반 게임 루프는 프레임 처리 시점이 아닐 때 `app_sleep(1)`을 호출하여 OS에 실행을 양보한다.

```C
// 게임 루프
for (;;)
{
    uint32_t now = app_get_ticks();
    // 경과된 Tick 계산
    uint32_t elapsed_ticks = (uint32_t)(now - last_frame_tick);
  
    // 아직 다음 frame을 처리할 tick이 되지 않았다면 CPU를 양보
    if (elapsed_ticks < PONG_FRAME_TICKS)
    {
        app_sleep(1);
        continue;
    }
  
    // 지연이 누적되었을 때 여러 frame을 몰아서 처리하지 않고,
    // 현재 tick 기준으로 다음 frame을 진행
    last_frame_tick = now;
  
    // Delta Ticks 클램핑
    uint32_t delta_ticks = pong_clamp_delta_ticks(elapsed_ticks);
  
    pong_handle_input(&game, delta_ticks);
    pong_update(&game, delta_ticks);
    pong_render(&game);
  
    app_sleep(1);
}
```

## 적용한 최적화 기법  
  
> 현재 적용된 최적화 기법은 **프레임 대기 구간에서 busy wait를 피하고 `SYS_SLEEP`을 통해 OS에 실행을 양보하는 정책**이다.
> 
> 이는 렌더링 자체의 비용을 줄이는 것이 아니라, 프레임 간 유휴 시간을 효율적으로 관리함으로써 **시스템 자원 분배를 최적화하고 가용성을 높이는 데 목적이 있다.**

# 분석

## 결과 해석

> `SYS_SLEEP`을 사용하는 조건에서는 `G=99`, `S=99`가 관측되었다. 이는 게임이 프레임 처리 시점이 아닐 때 OS Tick을 확인한 뒤, `SYS_SLEEP`을 호출하여 OS에 실행을 양보하고 있음을 의미한다.
> 
> 반면 Busy Wait 조건에서는 `G` 값이 폭증하고 `S` 값이 0이 되는 현상이 관측되었다. 이는 프레임 처리 시점이 아님에도 게임 루프가 계속 실행되며, 다음 frame 처리 가능 여부를 확인하기 위해 `SYS_GET_TICKS`를 과도하게 호출하고 있음을 의미한다.
> 
> Busy Wait 조건에서 마우스 커서가 사실상 움직이지 않는 수준의 반응성 저하도 관찰되었다. 이는 게임 루프가 CPU 시간을 과도하게 점유하여 마우스 처리나 커서 갱신 같은 다른 OS 작업이 수행될 여지를 줄였기 때문으로 해석된다.

# 결론

## 실험 결론

> `SYS_SLEEP`을 사용하는 프레임 대기 방식은 프레임 처리 시점이 아닐 때 게임이 CPU를 계속 점유하지 않고 OS에 실행을 양보하도록 만든다.
> 
> Busy Wait 방식에서는 `S(SYS_SLEEP)`가 0이 되었고, `G(SYS_GET_TICKS)` 호출 수가 폭증하였다.
> 이는 게임이 실제 프레임을 효율적으로 처리하는 것이 아니라, 다음 프레임 시점을 확인하기 위해 OS Tick을 과도하게 polling하고 있음을 의미한다.

## Overview

> 이번 실험에서는 게임 루프 내에서 다음 프레임 처리 시점까지 발생하는 유휴 시간을 관리하는 두 가지 정책을 비교하였다.
> 
> `SYS_SLEEP`을 사용하는 **양보(Yield) 정책**은 유휴 단계에서 CPU 제어권을 OS 스케줄러에 반환하는 반면, **Busy Wait 정책**은 루프를 지속하며 CPU 점유권을 유지하는 차이가 있다.
> 
> 자원 양보 정책의 부재는 단순한 수치 변화를 넘어 시스템 전체의 **가동성(Availability)** 저하로 직결되었다.
> Busy Wait 조건에서 관측된 마우스 커서의 반응성 마비는, 특정 프로세스가 CPU 타임 슬라이스를 독점할 때 커널이 마우스 처리와 같은 필수적인 시스템 태스크를 스케줄링하지 못하는 **기아 현상**을 단적으로 보여주는 예시이다.
> 
> 이번 실험을 통해 프레임 대기 구간에서의 적극적인 CPU 양보 정책은 애플리케이션의 성능 최적화를 넘어, **시스템 안정성과 사용자 경험(UX)을 보장하기 위한 필수적인 설계 요건**임을 확인할 수 있었다.
