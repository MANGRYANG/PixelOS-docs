- 실험 번호 : 01
- 관련 프로젝트 : `PixelOS`
- 실험 대상 모듈 :

# 서론
## 실험 목적

> Pong 렌더링의 기존 구조에서는, 중앙 기준선을 구성하는 여러 개의 사각형을 각각 `SYS_GAME_FILL_RECT` 시스템 콜을 반복 호출하는 방식으로 그린다.
> 
> 본 실험의 목적은 여러 사각형 렌더링 요청을 Batch 방식을 통해 하나로 묶어 처리함으로써 게임 애플리케이션과 OS 사이의 렌더링 시스템 콜 호출 횟수를 감소시킬 수 있는지 확인하는 것이다.

## 실험 범위

### 포함 범위

- 중앙선 렌더링에 사용되는 `fill_rect` 호출을 Batch 시스템 콜 하나로 묶는다.
- 화면에 보이는 중앙선의 위치, 크기, 간격, 색상은 변경하지 않는다.
- 시스템 콜 계측값을 통해 기존 구현과 최적화 후 구현을 비교한다.

### 제외 범위

- dirty rectangle 렌더링
- 배경 layer 캐싱
- 정적 UI 캐싱
- 공, 패들, 점수 렌더링 방식 변경
- 그래픽 품질 변경
- PS/2 입력 계층 수정

# 본론
## 실험 가설

> 중앙선을 구성하는 여러 개의 사각형을 개별 `SYS_GAME_FILL_RECT` 호출로 그리는 대신, 하나의 배치 시스템 콜로 전달하면 렌더링 관련 syscall 호출 수를 줄일 수 있다.

| 항목    | 내용                                                            |
| ----- | ------------------------------------------------------------- |
| 독립 변수 | 중앙선 렌더링 방식 (개별 `fill_rect` 호출 vs batch rect syscall)          |
| 종속 변수 | 프레임당 렌더링 syscall 호출 횟수, `SYS_GAME_FILL_RECT` 호출 횟수            |
| 통제 변수 | 창 크기, 공 속도, paddle 크기, 중앙선 개수/위치/색상, 입력 조건, 측정 시간 등           |
| 예상 결과 | 중앙선 segment 수가 `N`개일 때 프레임 당 렌더링 syscall 수가 약 `N - 1`만큼 감소한다. |

***

## 기존 구현에서의 문제점  
  
> 기존 구현에서는 중앙선의 각 segment를 `app_game_fill_rect()` 함수의 반복 호출을 통해 렌더링한다.
> 
> 해당 방식은 구현이 단순하지만, segment 개수가 늘어나게 되면 게임 애플리케이션에서 커널로 진입하는 syscall 횟수가 증가할 수 있다.

```C
__attribute__((section(".usertext")))
static void pong_render_center_line(PongState* game)
{
    int center_x = (game->game_w - CENTER_LINE_WIDTH) / 2;
    for (
	    int i = 0;
	    i < game->game_h;
	    i += (CENTER_LINE_HEIGHT + CENTER_LINE_GAP)
	)
    {
        app_game_fill_rect(
	        center_x,
	        i,
	        CENTER_LINE_WIDTH,
	        CENTER_LINE_HEIGHT,
	        COLOR_LIGHT_GRAY
	    );
    }
}
```

기존 구현에서는 렌더링 대상이 단순한 사각형 목록임에도, 각 사각형마다 유저 모드에서 커널 모드로의 전환이 반복되어 수행된다.

***
  
## 적용할 최적화 기법  
  
> 적용할 최적화 기법은 **렌더링 syscall batching**이다.
> 여러 개의 사각형 정보를 사용자 영역 배열에 구성한 뒤, 이를 하나의 시스템 콜로 커널에 전달한다.
> 커널은 syscall 한 번 안에서 배열을 순회하며 기존과 동일한 사각형들을 그린다.
  
---  
  
## 구현 방법 

| 변경 위치                  | 변경 내용                                                                                                                           |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `kernel/syscall.h`     | `SYS_GAME_FILL_RECTS_BATCH` syscall 번호, `SyscallRect` 구조체 추가                                                                    |
| `linker/linker.ld`     | `.userdata` 섹션 추가                                                                                                               |
| `kernel/kernel.c`      | `.userdata` 섹션에 `R/W`, `U/S` 플래그 설정                                                                                             |
| `app/app_api.h`        | `app_game_fill_rects_batch` wrapper 추가                                                                                          |
| `kernel/game_window.c` | 여러 사각형을 순회하며 그리는 `kernel_game_fill_rects_batch` 추가                                                                              |
| `kernel/interrupts.c`  | syscall handler에 `SYS_GAME_FILL_RECTS_BATCH` syscall 처리 케이스 추가<br><br>`SYS_GAME_FILL_RECTS_BATCH` 시스템 콜 계측 메시지 추가               |
| `app/pong.c`           | 중앙 기준선을 구성하는 사각형들을 저장하는 전역 배열인 `g_center_line_rects`를 `.userdata` 섹션 영역에 배치<br><br>중앙선 렌더링을 개별 `fill_rect` 반복 호출에서 Batch 호출로 변경 |

```C
// 기존 중앙선 렌더링 방식 (fill_rect 반복 호출)
__attribute__((section(".usertext")))
static void pong_render_center_line(PongState* game)
{
    int center_x = (game->game_w - CENTER_LINE_WIDTH) / 2;
    for (int i = 0; i < game->game_h;
	    i += (CENTER_LINE_HEIGHT + CENTER_LINE_GAP))
    {
        app_game_fill_rect(
	        center_x,
	        i,
	        CENTER_LINE_WIDTH,
	        CENTER_LINE_HEIGHT,
	        COLOR_LIGHT_GRAY
	    );
    }
}
```


```C
// 변경된 중앙선 렌더링 방식 (Batch 호출)
__attribute__((section(".usertext")))
static void pong_render_center_line(PongState* game)
{
    int center_x = (game->game_w - CENTER_LINE_WIDTH) / 2;
    uint32_t count = 0;
  
    for (int i = 0;
		    i < game->game_h && count < CENTER_LINE_MAX_RECTS;
		    i += (CENTER_LINE_HEIGHT + CENTER_LINE_GAP)
	    )
    {
        g_center_line_rects[count].x
	        = (uint16_t)center_x;
        g_center_line_rects[count].y
	        = (uint16_t)i;
        g_center_line_rects[count].width
	        = (uint16_t)CENTER_LINE_WIDTH;
        g_center_line_rects[count].height
	        = (uint16_t)CENTER_LINE_HEIGHT;
        ++count;
    }
  
    if (app_game_fill_rects_batch(
		    g_center_line_rects,
		    count,
		    COLOR_LIGHT_GRAY
		) < 0
	)
    {
        for (uint32_t i = 0; i < count; ++i)
        {
            app_game_fill_rect(
                g_center_line_rects[i].x,
                g_center_line_rects[i].y,
                g_center_line_rects[i].width,
                g_center_line_rects[i].height,
                COLOR_LIGHT_GRAY
            );
        }
    }
}

```

# 분석

## 결과 해석

> 실험 결과, 최적화 후에도 `P`, `K`, `G`, `S`, `T` 값은 거의 유지되었고, 개별 `SYS_GAME_FILL_RECT` 호출 수인 `F`는 frame 당 16회에서 3회로 감소하였다.
> 
> 추가로 batch syscall인 `B`가 frame 당 0회에서 1회 수준으로 증가하였다.
> 
> 이는 다수의 렌더링 Syscall을 하나의 Batch syscall로 대체하여 수행하였기 때문이다.

# 결론

## 실험 결론

> 중앙선 렌더링을 개별 `SYS_GAME_FILL_RECT` 호출 방식에서 `Batch syscall` 방식으로 변경하면, **화면에 표시되는 결과를 유지**하면서 **게임 애플리케이션과 OS 사이의 렌더링 syscall 호출 수를 감소**시킬 수 있다.
> 
> 본 실험은 PixelOS 프로젝트에서 draw call 감소를 통한 최적화를 **시스템 콜 경계 전환 감소** 관점에서 적용한 실험적 적용 모델이다.

## Overview

>  이번 실험은 **Draw call batching 최적화 기법**을 PixelOS의 syscall 기반 렌더링 구조에 맞게 재해석하여 적용하였음에 의의를 둔다.
